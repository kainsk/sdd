# L2 — BDD-сценарии

> **Статус:** детальный справочник
> **Шаблон:** [`templates/scenario.tpl.md`](../templates/scenario.tpl.md)
> **Уровни выше:** [L0 — Конституция](L0-constitution.md), [L1 — Доменная модель](L1-domain-model.md)
> **Связь с верхними уровнями:** каждый сценарий ОБЯЗАН опираться на сущности L1 и не противоречить L0.

---

## 1. Назначение

BDD-сценарии фиксируют **глаголы** системы: наблюдаемое поведение в терминах предусловий, действия и ожидаемого исхода. Если L1 описывает «что существует», то L2 описывает «что происходит».

L2 отвечает на три вопроса:

1. *Что система делает в ответ на конкретное действие?*
2. *Какой наблюдаемый исход свидетельствует об успехе?*
3. *Какие предусловия должны быть выполнены?*

L2 — это **поведение**, а не реализация. Как сценарий исполняется — вопрос кода.

---

## 2. Что сюда входит

| Категория | Описание |
|---|---|
| **Scenario / Сценарий** | Одно атомарное наблюдаемое поведение |
| **Feature / Возможность** | Группа сценариев, описывающих одну capability |
| **Background / Фон** | Общие предусловия для всех сценариев внутри Feature |
| **Scenario Outline / Таблица примеров** | Параметризованный сценарий с примерами |
| **Actor / Актор** | Кто инициирует действие (пользователь, другой сервис, планировщик) |

### Что сюда НЕ входит

- HTML-селекторы, названия кнопок, цвета → UI-тесты, не BDD
- SQL, имена таблиц, мок-объекты → интеграционные/юнит-тесты
- Алгоритмы, пороги производительности → L0 (бюджеты) или код
- Повторение инвариантов L1 — сценарий опирается на них, но не дублирует
- Non-behavioral требования без наблюдаемого исхода

## 3. Семантические поля

### 3.1 Метаданные документа

| Поле | Кратн. | Тип | Смысл |
|---|---|---|---|
| `version` | 1 | SemVer | Версия набора сценариев |
| `depends_on` | 1 | объект | `{ constitution: SemVer, domain: SemVer }` |
| `status` | 1 | enum | `draft` \| `active` \| `deprecated` — см. [Lifecycle в глоссарии](../glossary.md#lifecycle--жизненный-цикл-спецификации) |
| `feature_area` | 1 | string | Capability, которую группа сценариев покрывает |

### 3.2 Feature (группа сценариев)

| Поле | Кратн. | Тип | Описание |
|---|---|---|---|
| `id` | 1 | `FEA-NNN` | Стабильный ID возможности |
| `name` | 1 | string | Короткое имя capability |
| `narrative` | 1 | объект | `As a … I want … So that …` — бизнес-мотив |
| `background` | 0..1 | список `Given` | Общие предусловия |
| `scenarios` | 1..n | список `SCN-NNN` | Сценарии, входящие в Feature |

### 3.3 Сценарий

| Поле | Кратн. | Тип | Описание |
|---|---|---|---|
| `id` | 1 | `SCN-NNN` | Стабильный ID сценария |
| `title` | 1 | string | Короткое описание поведения |
| `feature` | 1 | `FEA-NNN` | К какой Feature относится |
| `priority` | 1 | enum | `MUST` \| `SHOULD` \| `MAY` — наследуется из RFC 2119 |
| `actors` | 1..n | список | Кто инициирует и кто наблюдает |
| `tags` | 0..n | список | Для фильтрации (`@smoke`, `@billing`, `@regression`) |
| `given` | 0..n | список | Предусловия |
| `when` | 1 | строка | **Ровно одно** действие |
| `then` | 1..n | список | Наблюдаемые исходы |
| `and_but` | 0..n | список | Дополнительные шаги (не заменяют `given`/`then`) |
| `references` | 1..n | объект | `{ entities: [ENT-NNN], invariants: [INV-NNN], articles: [ART-NNN] }` |
| `examples` | 0..1 | таблица | Для Scenario Outline |

**Критическое правило:** ровно один `when` на сценарий. Несколько действий — несколько сценариев.

### 3.4 Приоритет сценария

Приоритет наследует семантику RFC 2119 из L0:

- `MUST` — поведение обязательно; регрессия блокирует релиз
- `SHOULD` — сильно желательно; регрессия требует обоснования
- `MAY` — опционально; может быть отключено без эскалации

## 4. Правила написания

| Правило | Пояснение |
|---|---|
| **Один `when`** | Один сценарий — одно действие. Два действия — два сценария. |
| **Наблюдаемость** | `then` описывает то, что видно снаружи (API-ответ, событие, состояние), а не внутренности. |
| **Декларативность** | Формулируйте «что», а не «как». Не «кликает на кнопку», а «отправляет форму заказа». |
| **Терминология из L1** | Сущности, поля и состояния называйте дословно, как в доменной модели. |
| **Явные ссылки** | Каждый сценарий цитирует `ENT-NNN`, `INV-NNN` и `ART-NNN`, с которыми связан. |
| **Атомарность** | Один наблюдаемый эффект — один сценарий. Не группируйте несколько проверок. |
| **Независимость** | Сценарий не должен зависеть от порядка выполнения других сценариев. |
| **Determined outcome** | `then` не допускает формулировок «примерно», «как правило». |

---

## 5. Антипаттерны

- ❌ Несколько `when` в одном сценарии (`When I do X, And then I do Y`)
- ❌ UI-детали в `then`: «кнопка становится зелёной», «появляется тост»
- ❌ Утечка реализации: «строка добавлена в таблицу `orders`»
- ❌ Размытые исходы: «всё работает», «ошибки нет» (без указания наблюдаемого признака успеха)
- ❌ Сценарий без `when` — это просто состояние, а не поведение
- ❌ Сценарий, дублирующий инвариант L1 слово в слово (инвариант уже проверяется на уровне домена)
- ❌ Сценарии с именами, содержащими номер задачи или PR (теряют смысл со временем)
- ❌ Один огромный сценарий «happy path от регистрации до оплаты» — разбейте на атомарные
- ❌ Preconditions, описывающие внутреннее состояние БД

## 6. Минимальный пример (Gherkin-MD)

```markdown
---
version: 0.2.0
depends_on:
  constitution: 1.0.0
  domain: 0.3.0
status: active
feature_area: billing
---

## FEA-001 — Выставление счёта

**Narrative:**
- **As a** оператор биллинга
- **I want** переводить подготовленный счёт в статус `issued`
- **So that** контрагент получает обязательство к оплате

**Background:**
- Given существует `Counterparty` с идентификатором `C-100`
- And существует `Invoice` в статусе `draft` с номером `INV-2026-000001`

---

### SCN-001 — Успешное выставление непустого счёта

- **priority:** MUST
- **actors:** [billing-operator]
- **tags:** [@smoke, @billing]
- **references:**
  - entities: [ENT-001, ENT-002]
  - invariants: [INV-001, INV-002]
  - articles: [ART-003]

```gherkin
Given Invoice INV-2026-000001 has at least one InvoiceLine
When the operator issues Invoice INV-2026-000001
Then Invoice INV-2026-000001 status becomes "issued"
And a domain event InvoiceIssued is emitted with the invoice number and total
And no outbound network call is performed
```

---

### SCN-002 — Отказ выставить пустой счёт

- **priority:** MUST
- **actors:** [billing-operator]
- **tags:** [@billing, @negative]
- **references:**
  - entities: [ENT-001]
  - invariants: [INV-001]
  - articles: []

```gherkin
Given Invoice INV-2026-000001 has zero InvoiceLines
When the operator issues Invoice INV-2026-000001
Then the issue action is rejected with reason "empty-invoice"
And Invoice INV-2026-000001 status remains "draft"
And no domain event is emitted
```

---

### SCN-003 — Отказ выставить счёт со смешанными валютами

- **priority:** MUST
- **actors:** [billing-operator]
- **tags:** [@billing, @negative]
- **references:**
  - entities: [ENT-001, ENT-002, ENT-010]
  - invariants: [INV-003]
  - articles: []

```gherkin
Given Invoice INV-2026-000001 has two InvoiceLines in different currencies
When the operator issues Invoice INV-2026-000001
Then the issue action is rejected with reason "mixed-currency"
And Invoice INV-2026-000001 status remains "draft"
```
```

## 7. Scenario Outline (таблица примеров)

Используйте Outline, когда поведение одинаково, а меняются только данные. Это сокращает дублирование, но **не** заменяет атомарность: каждая строка примеров — это один логический сценарий.

```gherkin
Scenario Outline: Отказ при невалидной сумме позиции
  Given a new InvoiceLine with quantity <qty> and unitPrice <price>
  When the system validates the line
  Then the validation rejects it with reason "<reason>"

  Examples:
    | qty  | price | reason         |
    | 0    | 10.00 | non-positive   |
    | -1   | 10.00 | non-positive   |
    | 1    | -5.00 | negative-price |
```

**Правило:** колонки таблицы — это параметры, а не разные поведения. Если строки требуют разного `then` — разделите на отдельные сценарии.

## 8. Чеклист перед коммитом

- [ ] `version`, `depends_on.constitution`, `depends_on.domain`, `status`, `feature_area` заполнены
- [ ] Каждая Feature имеет `narrative` в формате As/I want/So that
- [ ] У каждого сценария **ровно один** `when`
- [ ] Каждый `then` описывает наблюдаемый исход, а не внутренности
- [ ] У каждого сценария проставлен `priority` (MUST/SHOULD/MAY)
- [ ] `references` содержит хотя бы одну ссылку на `ENT-NNN` или `INV-NNN`
- [ ] Сценарии независимы друг от друга
- [ ] Нет UI-деталей, SQL, имён таблиц
- [ ] Имена сущностей и полей совпадают с L1 дословно
- [ ] ID `FEA-NNN`, `SCN-NNN` уникальны и не переиспользованы
- [ ] Выполнен **Dry Run**: сценарии сверены с актуальными версиями L0 и L1

---

## 9. Как L2 используется в промптах

L2 внедряется срезом: только сценарии, относящиеся к задаче (обычно по `tags` или `feature_area`). Полный набор сценариев в промпт не помещается и не нужен.

Модели даётся инструкция:

> «L2 — твой эталон поведения. Не придумывай новых исходов. Если задача требует поведения, не описанного в L2 — ОСТАНОВИСЬ и сообщи, что нужен новый сценарий. Код ОБЯЗАН удовлетворять все цитируемые `then`; если `priority: MUST`, провал хотя бы одного — блокер.»

Стратегия срезов — см. [`03-context-injection.md`](../03-context-injection.md).
