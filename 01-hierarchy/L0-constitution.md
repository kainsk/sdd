# L0 — Конституция

> **Статус:** детальный справочник
> **Шаблон:** [`templates/constitution.tpl.md`](../templates/constitution.tpl.md)
> **Уровень ниже:** [L1 — Доменная модель](L1-domain-model.md)

---

## 1. Назначение

Конституция фиксирует **неизменные правила уровня проекта**. Это высший источник истины; каждая доменная модель (L1) и BDD-сценарий (L2) ОБЯЗАНЫ ей соответствовать.

Конституция отвечает на два вопроса:

1. *Что НИКОГДА не меняется без осознанной поправки?*
2. *Что побеждает при конфликте двух спецификаций?*

---

## 2. Что сюда входит

| Категория | Примеры |
|---|---|
| **tech-stack** | язык и его версия, рантайм, пин фреймворков |
| **architecture** | слои, направление зависимостей, порты и адаптеры |
| **security** | требования к authn/authz, минимумы по криптографии, обращение с секретами |
| **performance** | бюджеты latency/throughput, потолки памяти |
| **quality** | минимум покрытия тестами, линтерные гейты, политика ревью |
| **process** | ветвление, релизный цикл, правила внесения поправок |

### Что сюда НЕ входит

- Конкретные формы сущностей → L1
- Пользовательские сценарии, критерии приёмки → L2
- Детали реализации (пути файлов, имена классов) → код
- Субъективные цели («быстро», «чисто») без измеримых границ

## 3. Семантические поля

Документ Конституции ОБЯЗАН содержать перечисленные ниже поля. Кратность: `1` — ровно одно, `0..1` — опционально, `1..n` — одно или более.

### 3.1 Метаданные

| Поле | Кратн. | Тип | Смысл |
|---|---|---|---|
| `version` | 1 | SemVer | Версия Конституции |
| `effective_date` | 1 | ISO 8601 | Дата вступления в силу |
| `supersedes` | 0..1 | SemVer | Какую версию заменяет |
| `ratified_by` | 1..n | string | Кто утвердил (роли или имена) |
| `status` | 1 | enum | `draft` \| `active` \| `deprecated` — см. [Lifecycle в глоссарии](../glossary.md#lifecycle--жизненный-цикл-спецификации) |

### 3.2 Преамбула

1–3 предложения: **что за проект** и **зачем нужна Конституция**. Свободный текст, правил здесь нет — только контекст.

### 3.3 Глоссарий

Канонические определения терминов, используемых в статьях. Одно определение на термин; при расхождении в L1/L2 побеждает определение из L0.

### 3.4 Статьи

Каждая статья **атомарна** — одно правило, одна статья. Схема:

| Поле | Кратн. | Тип | Описание |
|---|---|---|---|
| `id` | 1 | `ART-NNN` | Стабильный идентификатор. Никогда не переиспользуется, даже после удаления |
| `title` | 1 | string | Короткий заголовок для человека |
| `category` | 1 | enum | См. §2 |
| `statement` | 1 | string | Сам текст правила. ОБЯЗАТЕЛЬНО ключевое слово RFC 2119 |
| `rationale` | 1 | string | Зачем правило существует: какой риск или цель оно адресует |
| `scope` | 1 | string | Где применяется (весь код / пакет X / рантайм / граница Y) |
| `exceptions` | 1 | string \| `none` | Явные исключения |
| `enforcement` | 1 | string | Как выявляется нарушение (линтер, CI, arch-test, пункт ревью) |
| `precedence` | 1 | enum | `critical` \| `normal` |
| `added_in` | 1 | SemVer | Версия, в которой статья появилась |

**Ключевые слова RFC 2119:**

- `MUST` / `MUST NOT` — абсолютное требование
- `SHOULD` / `SHOULD NOT` — настоятельная рекомендация с документируемыми исключениями
- `MAY` — действительно опционально

### 3.5 Правила старшинства

Явный порядок разрешения конфликтов:

1. L0 старше L1 и L2.
2. Внутри L0 `critical` старше `normal`.
3. Неоднозначности **ОБЯЗАНЫ эскалироваться**, а не угадываться моделью.

### 3.6 Процедура внесения поправок

Кто может менять Конституцию, по какой процедуре утверждения и как бампается версия:

- **Patch (`x.y.Z`)** — уточнения, опечатки, правки обоснования. Семантика не меняется.
- **Minor (`x.Y.0`)** — добавлены новые статьи. Существующие не тронуты.
- **Major (`X.0.0`)** — существующее правило изменено или удалено. Триггерит ре-ревью всех L1/L2.

## 4. Правила написания

| Правило | Пояснение |
|---|---|
| **Атомарность** | Одно правило — одна статья. |
| **Проверяемость** | Соответствие можно проверить механически (человеком на ревью или CI). |
| **Стабильность** | Любое изменение бампает версию; молча редактировать нельзя. |
| **ЧТО, а не КАК** | Описывается ограничение, а не реализация. |
| **RFC 2119** | В каждой формулировке есть MUST / SHOULD / MAY. |
| **Измеримость** | Прилагательные заменяются числами там, где возможно. |

---

## 5. Антипаттерны

- ❌ *«Код должен быть чистым и поддерживаемым.»* — не тестируемо
- ❌ *«Используйте лучшие практики обработки ошибок.»* — размыто
- ❌ Два правила в одной статье
- ❌ Конкретные пути файлов и имена классов
- ❌ Описание пользовательских сценариев — это уровень L2
- ❌ *«Быстро»*, *«просто»*, *«масштабируемо»* без числовых границ

## 6. Минимальный пример

```yaml
version: 1.0.0
effective_date: 2026-04-10
ratified_by: [tech-lead]
status: active

preamble: |
  CLI-инструмент для обработки PDF-счетов. Конституция охраняет
  гарантии offline-first и гексагональную границу.

glossary:
  Port: "Интерфейс, объявленный в пакете `ports` и принадлежащий доменному слою."
  Adapter: "Конкретная реализация порта, живущая вне доменного слоя."

articles:
  - id: ART-001
    title: Язык реализации
    category: tech-stack
    statement: "Проект MUST использовать Go версии 1.22 или новее как единственный язык реализации."
    rationale: "Экспертиза команды; целевой формат деплоя — статический бинарник."
    scope: весь исходный код
    exceptions: none
    enforcement: CI-проверка директивы toolchain в go.mod
    precedence: critical
    added_in: 1.0.0

  - id: ART-002
    title: Гексагональная граница
    category: architecture
    statement: "Доменные пакеты MUST NOT импортировать пакеты адаптеров."
    rationale: "Сохраняет доменный слой тестируемым без моков."
    scope: /internal/domain/**
    exceptions: none
    enforcement: go-arch-lint, правило `domain-no-adapters`
    precedence: critical
    added_in: 1.0.0

  - id: ART-003
    title: Сетевой egress
    category: security
    statement: "Бинарник MUST NOT совершать исходящие сетевые вызовы в рантайме."
    rationale: "Гарантия offline-first; требования к резидентности данных."
    scope: поведение в рантайме
    exceptions: "Единственная opt-in команда `--check-update` MAY обращаться к серверу релизов."
    enforcement: integration-тест в сетевом namespace с отключённой сетью
    precedence: critical
    added_in: 1.0.0

precedence_rules:
  - "L0 старше L1 и L2."
  - "Внутри L0 `critical` старше `normal`."
  - "Неоднозначности ОБЯЗАНЫ эскалироваться тех-лиду."

amendment_procedure: |
  Поправки приезжают как PR с меткой `constitution`. Major-бамп требует
  двух апрувов мейнтейнеров и миграционного примечания для L1/L2-спек.
```

## 7. Чеклист перед коммитом

- [ ] `version`, `effective_date`, `status` заполнены
- [ ] Каждая статья атомарна (ровно одно правило)
- [ ] В каждой формулировке есть ключевое слово RFC 2119
- [ ] У каждой статьи непустое `rationale`
- [ ] У каждой статьи указан `enforcement` — механическая проверка или оформленный шаг ревью
- [ ] Нет деталей реализации
- [ ] Глоссарий покрывает все неочевидные термины, встречающиеся в формулировках
- [ ] Правила старшинства заданы явно
- [ ] Процедура внесения поправок описана

## 8. Как L0 используется в промптах

Конституция внедряется **дословно** в начало system prompt, перед L1 и L2. Она pinned: никогда не урезается и не суммаризируется.

Модели даётся инструкция:

> «Считай L0 абсолютным законом. Перед выдачей кода или плана сверь запрос с L0. При конфликте ОСТАНОВИСЬ и сообщи. Не пытайся одновременно удовлетворить противоречивые требования.»
