# Золотые правила SDD

Четыре правила, которые ломать нельзя. Остальное — тактика.

---

## Rule 1 — Atomic Specs / Атомарность

**Правило:** одна спецификационная единица = одно утверждение.

- Одна статья L0 — одно правило
- Одна сущность L1 — один блок; один инвариант — один `INV-NNN`
- Один сценарий L2 — одно действие (`when`) и одно наблюдаемое поведение

### Зачем

- **Точечные ссылки.** `INV-003` можно процитировать; «третий абзац секции 2.1» — нет.
- **Стабильные правки.** Изменение одного правила не трогает соседние, не триггерит ре-ревью связанных инвариантов.
- **Механическая проверяемость.** Атомарное правило легко превратить в линтерное правило, refine-блок или ассерт. Комбинированное — нельзя.
- **Борьба с review fatigue.** Атомарные спеки можно ревьюить диффом: «добавили INV-007, изменили ART-002». Монолитный Markdown превращается в нечитаемую стену.

### Тест на атомарность

Если в формулировке есть `и` / `AND`, связывающее два самостоятельных требования — правило не атомарно. Разбейте.

Пример:

- ❌ `ART-042: The API MUST reject requests without auth AND log every rejection.`
- ✅ `ART-042: The API MUST reject unauthenticated requests.`
- ✅ `ART-043: The API MUST log every rejected request at level WARN.`

### Конкретные численные лимиты

Атомарность — абстрактный принцип. Чтобы сделать его проверяемым, у каждого уровня есть **числовые границы**. Превышение — запах, что единица не атомарна.

| Уровень | Единица | Мягкий лимит | Жёсткий лимит |
|---|---|---|---|
| L0 | 1 статья | ~10 строк prose | при превышении — разбить |
| L1 | 1 сущность | ~15 полей | больше = переоценить bounded context |
| L1 | 1 инвариант | 1 предложение с `MUST` | любое перечисление — разбить |
| L2 | 1 сценарий | **3–7 шагов** | **≤ 10 шагов** |
| L2 | `when` на сценарий | — | **ровно 1** |
| L2 | `Background` | — | **≤ 4 строки** |
| L2 | сценариев на feature | — | **≤ 12** |
| core patterns | `memory/patterns.md` | 80% (56 строк / 4.4 KB) | **70 строк / 5.5 KB** |
| satellite patterns | `memory/patterns/*.md` | 25 строк | 30 строк (soft) |

Численные лимиты L2 заимствованы из канонических BDD-best-practices (Cucumber, Automation Panda) — см. [`02-gherkin-md.md` §8](02-gherkin-md.md#8-лимиты-и-стиль). Лимиты patterns — из cog-механики, см. [`06-patterns.md`](06-patterns.md).

## Rule 2 — Single Source of Truth / Единственный источник правды

**Правило:** у каждого факта ровно одно место, где он определён. Всё остальное — ссылки.

### Иерархия правды

```
L0 Constitution         ← правда о проекте в целом
   │
   │  (ссылается вниз через enforcement / область действия)
   │
L1 Domain Model         ← правда о форме и инвариантах
   │
   │  (ссылается вверх через constraints_from_L0)
   │  (ссылается вниз через references в сценариях)
   │
L2 BDD Scenarios        ← правда о поведении
   │
   │  (ссылается вверх через references.entities / .invariants / .articles)
   │
Code / Tests            ← исполнение правды
   │
   │  (цитирует ENT-NNN / INV-NNN / SCN-NNN в комментариях заголовка)
```

Правила старшинства:

1. При конфликте выигрывает уровень выше
2. Внутри уровня `critical` > `normal`
3. Неоднозначность эскалируется, не угадывается

### Запреты дублирования

- ❌ Инвариант, скопированный из L1 в сценарий L2 — сценарий ссылается, не цитирует
- ❌ Константа бюджета производительности в L0 **и** в сценарии L2 — живёт только в L0
- ❌ Описание сущности в prose-README и в L1 — README ссылается на `ENT-NNN`, не пересказывает
- ❌ Два файла с разными версиями одного и того же `ART-NNN`
- ❌ Инвариант, выраженный и как `INV-NNN`, и как тест — тест цитирует `INV-NNN` в имени/комментарии

### Тест SSOT

Измените один факт — сколько файлов придётся отредактировать? Если больше одного — SSOT нарушен.

## Rule 3 — Dry Run / Сверка перед кодингом

**Правило:** перед генерацией любого кода или плана модель обязана пройти Dry Run и сообщить о противоречиях.

### Минимальный протокол

```text
Перед генерацией кода:
1. Перечисли каждый ART/ENT/INV/SCN, которого касается задача.
2. Проверь каждый на противоречие с задачей.
3. Если есть конфликт с L0 — STOP. Процитируй конфликтующее правило.
4. Если требуется сущность/сценарий, отсутствующий в спеках — STOP. Запроси спеку.
5. Только после чистого Dry Run — код.
```

### Что ловит Dry Run

- **False control.** Большой контекст ≠ соблюдение спек. Модель может «прочитать» L0 и всё равно нарушить — Dry Run заставляет её явно сослаться на каждую статью.
- **Пропущенные спеки.** Если сущности нет в L1, но код её требует — это повод добавить `ENT-NNN`, а не выдумать поле.
- **Версионный дрейф.** Dry Run сверяет `depends_on` с фактической версией L0/L1.
- **Неоднозначность.** Термин без определения в глоссарии всплывает в отчёте.

### Что Dry Run **не** делает

Dry Run не гарантирует корректность кода. Он гарантирует только, что модель **знает**, с какими спеками работает, и не противоречит им явно. Всё остальное — юнит-тесты, интеграционные тесты, review.

### Интеграция с Context Injection

Dry Run — это *правило*. Как именно сверка встраивается в system prompt и как модель получает версионный handshake — в [`03-context-injection.md` → §5-7](03-context-injection.md#5-скелет-system-prompt).

### Формат вывода

Первый блок любого ответа модели на задачу:

```markdown
## Dry Run
- Touched: ART-001, ART-003, ENT-001, INV-001, INV-002, SCN-001, SCN-002
- Contradictions: none
- Missing specs: none
- Version handshake: constitution=1.0.0, domain=0.3.0, scenarios=0.2.0 ✓

## Plan
...

## Code
...
```

Если секции `Dry Run` нет — ответ не принимается.

## Rule 4 — Stable Identifiers / Стабильные идентификаторы

**Правило:** `ART-NNN`, `ENT-NNN`, `INV-NNN`, `EVT-NNN`, `FEA-NNN`, `SCN-NNN` — вечные. Никогда не переиспользуются, даже после удаления.

### Зачем

- Комментарии в коде, commit-месседжи, PR-описания, дашборды, алерты — всё ссылается на ID. Переиспользование ID = тихое изменение смысла ссылки.
- История изменений читается: «INV-003 удалён в v2.0.0» остаётся понятным навсегда.
- Трассировка регрессий: тест с именем `TestInvoice_INV_003_mixed_currency` однозначно сопоставляется с инвариантом.

### Правила

1. **Номера монотонно возрастают.** Никаких «заполним дырку от удалённого».
2. **Удалённые ID помечаются `status: deprecated`** с датой и причиной, а не удаляются из документа целиком (опционально, но сильно помогает при историческом ревью).
3. **ID не содержит семантики.** `ART-017` не означает «семнадцатая по важности» — это просто номер.
4. **Префикс фиксирован.** `ART` только для статей Конституции, `ENT` только для сущностей, и так далее.

## Бонус: Maturity Ladder / Зрелостная шкала

Не все проекты должны одновременно сидеть на одном уровне зрелости спек. Полезно понимать, где вы находитесь и куда целитесь:

| Уровень | Признаки | Риски |
|---|---|---|
| **Spec-first** | Спеки пишутся под задачу и выкидываются после мёржа | Дрейф между спеками и кодом гарантирован |
| **Spec-anchored** | Спеки живут в репо, версионируются, обновляются при изменениях | Дрейф возможен; нужна дисциплина ревью |
| **Spec-as-source** | Код помечен `DO NOT EDIT`, редактируются только спеки, код генерируется | Высокая цена входа; детерминизм LLM — узкое место |

Практический совет: большинству команд стоит целиться в **spec-anchored** и удерживать его. Spec-as-source — это отдельный инженерный проект, а не конфигурация. Spec-first — это прототипирование, не SDD.

SDD-справочник в этом репо описывает процесс, который начинает работать со спек-anchored.

## Четыре проверки перед мёржем

- [ ] **Atomic** — каждое новое правило / сущность / инвариант / сценарий атомарно
- [ ] **SSOT** — изменение одного факта требует правки ровно одного файла
- [ ] **Dry Run** — любой сгенерированный код сопровождён Dry Run-отчётом
- [ ] **Stable IDs** — все новые ID монотонно возрастают, удалённые не переиспользованы

## Гейты — таксономия проверок

Все механизмы остановки в проекте — Dry Run BLOCK, validate ERROR, housekeeping warnings, install-hooks pre-commit — это **гейты**. Без единой модели они выглядят разрозненными правилами; с моделью становятся осознанной системой защиты.

Заимствовано из [GSD](https://github.com/gsd-build/get-shit-done) v1.34.0 «gates taxonomy», адаптировано под наш набор инструментов.

### Четыре канонических типа

| Тип | Когда срабатывает | Что делает | Кто решает |
|---|---|---|---|
| **Pre-flight** | до начала фазы | блокирует старт | автоматический инструмент |
| **Revision** | в середине исполнения | возвращает на доработку | автоматический инструмент или ревьюер |
| **Escalation** | при неоднозначности или конфликте | передаёт человеку, не решает сама | модель / инструмент |
| **Abort** | при критическом нарушении | останавливает всю работу | автоматический инструмент |

### Маппинг существующих гейтов проекта

| Механизм | Тип | Когда |
|---|---|---|
| `/sdd:dry-run` Verdict: BLOCK | **Pre-flight** | До генерации кода: задача противоречит L0/L1/L2 |
| `/sdd:dry-run` Missing specs | **Escalation** | Задача требует ENT/SCN, отсутствующих в спеках — нужно решение человека о добавлении |
| `/sdd:dry-run` Ambiguous terms | **Escalation** | Термин не определён в глоссарии — нужно решение, как его трактовать |
| `sdd-validate.sh` ERROR | **Pre-flight** (через pre-commit hook) | До коммита: дублированные ID, version mismatch, превышение hard cap patterns |
| `sdd-validate.sh` WARN | **Revision** | Не блокирует, но предлагает доработку: orphan entities, отсутствие RFC 2119, soft cap satellites |
| `/sdd:housekeeping` errors | **Pre-flight** перед релизом | Невалидное состояние спек блокирует release |
| `/sdd:bug` шаг 2 (Analyze) — спека не найдена | **Abort** этого пайплайна | «Это не баг, это новая фича» — переключиться на полный `/sdd:dry-run` |
| Hard cap превышен на core patterns | **Pre-flight** | Блокирует через `sdd-validate.sh` exit 1 |
| Soft cap превышен на satellites | **Revision** | WARN — допустимо в dev, ревью попросит конденсировать |
| `/sdd:evolve` proposal | **Revision** уровня rules | Не меняет правила, предлагает изменения для обсуждения |
| RFC 2119 отсутствует в `ART-NNN` | **Revision** WARN | Допустимо для draft, обязательно для active |

### Зачем таксономия

1. **Словарь.** Когда новый разработчик спрашивает «почему validate ругается?» — ответ структурирован: «это revision gate, не блокер; устрани при возможности».
2. **Дизайн новых проверок.** При добавлении проверки сразу решаешь её тип: pre-flight (блокер) или revision (warn). Это форсирует осознанность.
3. **Эскалация ясна.** Escalation gates явно отмечены — модель не пытается их «обойти», а возвращает решение человеку.
4. **Соответствие фазам жизненного цикла.** Pre-flight — на старте, revision — в процессе, abort — на критическом нарушении, escalation — на любом этапе при неоднозначности.

### Правила для новых гейтов

При добавлении новой проверки в `sdd-validate.sh`, новой команды, нового pattern:

1. Явно классифицировать гейт по таксономии (комментарием в коде или в `command.md`).
2. **Pre-flight и Abort** — `exit 1` или явная остановка модели; никаких компромиссов.
3. **Revision** — `WARN` или текстовая рекомендация; не блокирует.
4. **Escalation** — обязательно содержит вопрос или choice points для человека; модель никогда не решает сама.
5. Не смешивать типы в одной проверке. Если гейт одновременно блокер и предлагает доработку — это два разных гейта.

## Механизация правил: Pattern Routing

Четыре правила выше — это **что** должно быть истинно в спеках. Как Claude обеспечивает эту истинность в каждом ответе — это отдельный вопрос: он решается через **pattern routing**.

Операционные правила работы Claude со спеками живут в `memory/patterns.md` (core, универсально) и `memory/patterns/<context>.md` (satellite, доменно-специфично). Core загружается каждой командой до спек; satellites — только когда задача касается соответствующего bounded context.

Это механизация Правила 2 (SSOT) на уровне operational rules: вместо того, чтобы дублировать инструкции по всем `commands/*.md` и system prompt'ам, правила хранятся в одном месте и цитируются ссылкой.

Hard cap 70 строк / 5.5 KB на core принудительно удерживает правила атомарными (Правило 1). Валидация через `bin/sdd-validate.sh`.

Полный протокол — [`06-patterns.md`](06-patterns.md).
