# /sdd:foresight

Nudge-mode аудит. Подсвечивает сигналы конвергенции и слепые пятна в спеках. **Диагностика, не лечение.**

Отличается от `/sdd:housekeeping`:
- housekeeping проверяет целостность (ошибки и warnings)
- foresight ищет паттерны и тенденции

Отличается от `/sdd:evolve`:
- evolve меняет правила
- foresight работает внутри существующих правил

## Когда применять

- Перед мажорным бампом L0
- Ежемесячно на активных проектах
- Когда housekeeping стабильно зелёный, а ощущение «что-то не так» остаётся
- Перед планированием следующего спринта по домену

## Шаги

1. Запустить `sdd foresight specs` (или `sdd foresight specs`).
2. Прочитать отчёт. Четыре секции:
   - **Hot invariants** — инварианты, процитированные 3+ файлами. Высокий blast-radius.
   - **Hot articles** — статьи Конституции, цитируемые из 2+ файлов. Архитектурные hotspots.
   - **Dormant entities** — сущности без coverage в сценариях.
   - **Thin scenario coverage** — feature с одним сценарием или без.
   - **Priority distribution** — отношение MUST/SHOULD/MAY.
3. Сформулировать один nudge для пользователя:
   - Одно наблюдение
   - Одно конкретное предложение действия
   - Цитата из отчёта как обоснование
4. Передать решение человеку. **Не применять изменения автоматически.**

## Файлы

- **Читает:** `specs/**`
- **Пишет:** ничего (отчёт в stdout; можно перенаправить в `specs/foresight.md`, если нужна история)
- **Запускает:** `bin/sdd-foresight.sh`

## Правила

- **Одно наблюдение, один nudge.** Не перечислять все находки — это утомляет; выбрать одну самую острую.
- **Не путать корреляцию с причинностью.** Foresight показывает ассоциации, не причины.
- **Dormant entities не автоматически ошибка.** Может быть сущность, которая есть в L1 для структурной связности, но не требует прямого coverage сценариями.
- **Не вызывать часто.** Сигналы foresight медленно меняются; еженедельный прогон — максимум.
