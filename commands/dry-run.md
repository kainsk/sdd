# /sdd:dry-run

Обязательная сверка задачи со спецификациями. Никакой код не может быть сгенерирован до чистого прогона этой команды.

## Когда применять

Всегда, когда пользователь просит код, затрагивающий домен, поведение или архитектурные границы. Пропускать можно только для тривиальных изменений, перечисленных в [`03-context-injection.md` → §10](../03-context-injection.md#10-когда-sdd-не-применять).

## Шаги

1. **Загрузить pattern-routing (до спек):**
   - Core: `memory/patterns.md` — всегда, дословно
   - Satellites: `memory/patterns/<context>.md` для каждого bounded context, упомянутого в задаче (grep имён context в задаче или путях спек-файлов). Если satellite отсутствует — не ошибка, работать только с core.
   - См. [`06-patterns.md`](../06-patterns.md) для полного протокола routing.

2. **Загрузить бандл спеков:**
   - L0: `specs/constitution.md` — дословно, без сжатия
   - L1: `specs/domain/*.md`
   - L2: `specs/scenarios/*.md`
   - Глоссарий: `specs/glossary.md` или глоссарий внутри Конституции

3. **Версионный handshake.** Запустить `sdd validate specs` и убедиться, что `depends_on.constitution` во всех L1/L2 совпадает с версией L0. Если не совпадает — СТОП.

4. **Построить touched-set.** Прочитать задачу пользователя и собрать:
   - Явные упоминания сущностей по имени → `ENT-NNN`
   - Упомянутые действия/процессы → сопоставить с `SCN-NNN` через `feature_area` и `tags`
   - Транзитивное замыкание: `relations.target` упомянутых сущностей, `references.articles` упомянутых сценариев

5. **Проверить противоречия:**
   - Для каждой статьи `ART-NNN` с `precedence: critical`, затронутой задачей — нарушает ли задача формулировку?
   - Для каждого инварианта `INV-NNN` — ломает ли задача инвариант?
   - Для каждого сценария `SCN-NNN` с `priority: MUST` — сохраняет ли задача все его `Then`?

6. **Проверить полноту:**
   - Задача упоминает сущность, отсутствующую в L1? → СТОП, требуется новая `ENT-NNN`.
   - Задача описывает поведение без покрытия в L2? → СТОП, требуется новый `SCN-NNN`.
   - Термин задачи не определён в глоссарии и неоднозначен? → СТОП, запросить уточнение.

7. **Вывести отчёт в точном формате:**

   ```markdown
   ## Dry Run

   - **Version handshake:** constitution=<X>, domain=<Y>, scenarios=<Z> ✓|✗
   - **Touched:** <список ART/ENT/INV/EVT/FEA/SCN через запятую>
   - **Contradictions:**
     - <none>
     - ИЛИ: `<ID>` — <дословная цитата правила, которое нарушается>
   - **Missing specs:**
     - <none>
     - ИЛИ: требуется `<ENT-NNN | SCN-NNN>` для <описание пробела>
   - **Ambiguous terms:**
     - <none>
     - ИЛИ: `<термин>` — <почему неоднозначно>
   - **Verdict:** `PROCEED` | `BLOCK`
   ```

8. **Если Verdict = BLOCK:** остановиться. Никакого кода. Ждать, пока пользователь разрешит конфликт (правка спек, эскалация, отказ от задачи).

9. **Если Verdict = PROCEED:** следующий ответ может содержать код. Секция `## Dry Run` остаётся в ответе первой.

10. **Append-only log.** Независимо от Verdict, добавить запись о прогоне в `specs/dry-run-log.md` (создать файл, если отсутствует). Формат — один YAML-документ на запись через `---`-разделитель:

   ```yaml
   ---
   run: 2026-04-11T12:34:56Z
   task_summary: "<одна строка — что делали>"
   touched: [ART-002, ART-003, ENT-001, INV-001, SCN-001]
   contradictions: 0
   missing: 0
   ambiguous: 0
   verdict: PROCEED
   ```

   Поля:
   - `run` — ISO 8601 timestamp UTC
   - `task_summary` — одна строка, ≤ 80 символов
   - `touched` — список затронутых ID
   - `contradictions` / `missing` / `ambiguous` — числа находок
   - `verdict` — `PROCEED` или `BLOCK`

   Этот лог append-only — никогда не редактируется. Его потом майнит [`/sdd:reflect`](sdd-reflect.md) для обнаружения повторяющихся блокеров и неоднозначных терминов.

## Файлы

- **Читает:** `memory/patterns.md`, `memory/patterns/**`, `specs/**`
- **Пишет:** `specs/dry-run-log.md` (append-only, после каждого прогона)

## Правила

- **Никогда не суммаризировать L0.** Цитаты — дословные.
- **Никогда не угадывать.** Неоднозначность → `Ambiguous terms`.
- **Никогда не продолжать после BLOCK** без явного согласия пользователя.
- **Никогда не пропускать Dry Run** «ради скорости». Это единственная защита от *false control*.
