# /sdd:evolve

Мета-аудит SDD-процесса. Отличается от `/sdd:housekeeping`: **housekeeping применяет правила; evolve меняет правила.**

## Когда применять

- Перед мажорным бампом Конституции
- Ежеквартально на активных проектах
- Когда housekeeping стабильно выдаёт один и тот же класс warnings — это сигнал, что правило само требует пересмотра

## Шаги

1. **Временное окно.** По умолчанию — последние 14 дней. Можно передать тег или дату: `/sdd:evolve --since v1.0.0`.
2. **Diff спек:** `git diff <since>..HEAD -- specs/` — понять, что изменилось.
3. **Подсчитать дельту:**
   - Новые статьи / сущности / инварианты / сценарии (добавлено)
   - Изменённые (modified)
   - Удалённые (deleted, с проверкой, что ID остались выжженными — не переиспользованы)
4. **Scorecard delta.** Сравнить `specs/scorecard.md` текущий vs предыдущий (через `git show`). Вычислить тренды.
5. **Запустить `/sdd:housekeeping`** и захватить текущий список warnings.
6. **Искать сигналы рассинхронизации правил и реальности:**
   - Статья, изменённая без version bump → нарушение процедуры
   - Версия Конституции стабильна, а L1/L2 меняются интенсивно → напряжение: либо спеки ниже противоречат Конституции, либо Конституция не покрывает важную территорию
   - Доминируют сценарии `priority: MAY` → спеки слишком мягкие
   - Стабильные orphan-сущности → доменный дрейф
   - Один и тот же warning неделями → правило не работает
7. **Предложить изменения правил, не контента:**
   - «Предложение: добавить в `04-golden-rules.md` правило — PR, добавляющий `ENT-NNN` без одновременного `SCN-NNN`, отклоняется»
   - «Предложение: поднять `precedence` для ART-007 с `normal` до `critical` — 3 сценария уже обходят его»
   - «Предложение: ввести `hard cap` на количество активных статей в категории `architecture` — сейчас 23»
   - **«Предложение: перенести pattern X из `memory/patterns.md` (core) в `memory/patterns/<context>.md` (satellite)»** — если pattern касается только одного bounded context
   - **«Предложение: добавить core pattern Y»** — если reflect нашёл 3+ повторяющихся операционных блокера
   - **«Предложение: удалить core pattern Z»** — если scorecard показывает, что он не работает (повторно нарушается, но не ловит регрессий)
   - См. [`06-patterns.md`](../06-patterns.md) для протокола routing и caps
8. **Записать в `specs/evolve-log.md`** (append-only):

   ```markdown
   ## <YYYY-MM-DD> evolve run

   **Window:** <since>..HEAD
   **Delta:** +<A> -<D> statements
   **Scorecard shift:** <headline numbers>
   **Signals:**
   - <observation 1>
   - <observation 2>
   **Proposed rule changes:**
   - <proposal 1>
   - <proposal 2>
   ```

9. **Отчитаться пользователю** коротким summary и попросить решение по proposals. Evolve не применяет изменения сам.

## Файлы

- **Читает:** `specs/**`, `memory/patterns.md`, `memory/patterns/**`, git history, `specs/scorecard.md` (текущий и предыдущий)
- **Пишет:** `specs/evolve-log.md` (append-only)

## Правила

- **Не меняет контент** — только правила. Контентные правки идут через обычный workflow редактирования спек.
- **Append-only лог** — прошлые записи evolve не переписываются, даже если предложения не были приняты. История решений ценна.
- **Предложения — не императивы.** Финальное слово за человеком.
