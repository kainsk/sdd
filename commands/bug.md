# /sdd:bug

Структурированный fast track для фикса бага. Альтернатива тяжёлому полному `/sdd:dry-run` для случаев, когда задача **не порождает новых ART/ENT/INV/SCN**, а только восстанавливает соответствие кода уже существующим спекам.

Заимствовано из bug fix workflow `Pimzino/claude-code-spec-workflow` (Report → Analyze → Fix → Verify) и адаптировано под нашу спек-модель.

## Когда применять

✅ **Использовать `/sdd:bug`, если:**

- Поведение наблюдается, но не совпадает с уже существующим `SCN-NNN` или нарушает уже существующий `INV-NNN`
- Инвариант/сценарий не нужно создавать заново — он уже описан, нарушение в реализации
- Изменение локализовано (1–3 файла кода)
- Нет архитектурных или контрактных изменений

❌ **Не использовать (нужен полный `/sdd:dry-run`), если:**

- Требуется новая сущность (`ENT-NNN`)
- Требуется новый инвариант (`INV-NNN`)
- Требуется новый сценарий (`SCN-NNN`)
- Изменение пересекает несколько bounded contexts
- Затронута статья Конституции (`ART-NNN`)

См. также: [`03-context-injection.md` → §10](../03-context-injection.md#10-когда-sdd-не-применять).

## Шаги

### 1. Report — что наблюдается

Зафиксировать в одной структуре:
- **Symptom:** что видит пользователь / что показывает тест
- **Expected:** какое поведение описано в `SCN-NNN` или `INV-NNN`
- **Repro:** минимальные шаги воспроизведения (≤ 5 строк)
- **Severity:** `critical` | `high` | `medium` | `low`

### 2. Analyze — какие спеки затронуты

- Загрузить core patterns (`memory/patterns.md`) — обязательно.
- Найти `SCN-NNN`/`INV-NNN`/`ART-NNN`, описывающий ожидаемое поведение. **Если не найден** — это не баг, это **новая фича**: переключиться на `/sdd:dry-run`.
- Найти место в коде, где поведение реализовано (или должно быть).
- Сформулировать **минимальный touched-set**: только те ID, которые цитируются исправлением.

### 3. Fix — изменение кода + тест

- Внести минимальное изменение в код. **Никакого рефакторинга по пути.**
- Добавить или восстановить тест, привязанный к `SCN-NNN`/`INV-NNN` (как именованный test case с ID в имени).
- Тест должен **падать до фикса** и **проходить после**.

### 4. Verify — регрессия не вернулась

- Запустить полный тест-набор (не только новый тест).
- Запустить `sdd validate specs` — спеки не изменились, валидация должна остаться зелёной.
- Если фикс затронул код, отвечающий за state machine или критичный invariant — добавить cross-check сценарий.

## Формат отчёта

После выполнения вывести в чат:

```markdown
## Bug Fix Report

- **Touched specs:** SCN-NNN, INV-NNN
- **Symptom:** <одна строка>
- **Root cause:** <одна-две строки>
- **Fix:** <изменённые файлы + 1 строка о сути>
- **Test:** <имя теста + статус "added/restored">
- **Verification:**
  - Test suite: PASS
  - Validate: PASS (0 errors)
  - Manual repro: cleared
```

## Файлы

- **Читает:** `memory/patterns.md`, `specs/**`, исходники
- **Пишет:** код проекта, тест, опционально `specs/dry-run-log.md` с verdict `PROCEED` и пометкой `bug-fix: true`

## Правила

- **Атомарность:** один баг — один прогон `/sdd:bug`. Если нашли два — два отдельных прогона.
- **Никакого scope creep.** Если по пути обнаружены другие баги — записать в issue tracker, не править в этом же фиксе.
- **Никаких новых ART/ENT/INV/SCN.** Если они нужны — это не баг. STOP, переключиться на `/sdd:dry-run`.
- **Тест обязателен.** Без падающего-до-фикса теста баг не считается зафикшенным (он вернётся).
- **Полная валидация перед коммитом.** `sdd test` (если бэг-фикс затронул скрипты) или хотя бы `sdd-validate.sh` на спеках проекта.
