# /sdd:scenario

Моделирование решения, затрагивающего спеки на мажорном уровне: переформулировка статьи Конституции, выделение нового bounded context, удаление спорной сущности, мажорный бамп.

Не для ежедневных правок — только для решений, у которых есть **точка невозврата** и **несколько веток исхода**.

## Когда применять

- Major bump L0 с ре-ревью всех L1/L2
- Разделение bounded context на два
- Удаление `ART-NNN`, на которую ссылается много сценариев
- Смена диалекта DSL (например, переход L2 с Markdown-таблиц на Gherkin-MD)
- Любое решение, где «пока ты выбираешь А, путь Б закрывается»

## Шаги

1. **Уточнить decision point.** Записать:
   - Вопрос: что именно решается одной фразой
   - Стейкс: что выигрывается/теряется в каждом исходе
   - Временная чувствительность: есть ли дедлайн, почему именно сейчас
   - Необратимость: насколько трудно откатить
2. **Dependency mapping:**
   - **Upstream constraints:** статьи L0, которые *ограничивают* решение
   - **Downstream consequences:** сущности/сценарии/код, которые изменятся в каждой ветке
3. **Построить 2–3 ветки.** Для каждой:
   - **Path:** что именно делается
   - **Timeline:** какие шаги, в каком порядке, за какое время
   - **Assumptions:** что должно быть истинно, чтобы ветка сработала
   - **Risks:** что может пойти не так
   - **Confidence:** subjective high/medium/low
4. **Timeline overlay:** наложить каждую ветку на реальный календарь — релизы, freeze-периоды, другие инициативы.
5. **Contingency mapping:** для каждой ветки — ранние сигналы (canary), которые скажут, что идёт не так, и что делать в этом случае.
6. **Записать решение** в `specs/scenarios/<slug>.md` (создать директорию при необходимости):

   ```markdown
   ---
   type: decision
   slug: <short-slug>
   created: <YYYY-MM-DD>
   deciding_by: <YYYY-MM-DD>
   status: open | decided | superseded
   ---

   # <Decision title>

   ## Question
   <одна фраза>

   ## Stakes
   - Выигрыш: <...>
   - Проигрыш: <...>

   ## Dependencies
   - Upstream: <ART-NNN, ART-NNN>
   - Downstream: <ENT-NNN, SCN-NNN, ...>

   ## Branches

   ### Branch A — <name>
   - Path: <...>
   - Timeline: <...>
   - Assumptions: <...>
   - Risks: <...>
   - Confidence: high|medium|low

   ### Branch B — <name>
   <...>

   ## Contingency map
   - Canary: <ранний сигнал> → <pivot action>

   ## Decision
   <если принято: какая ветка и почему; иначе — "open">
   ```

7. **Вывести summary** пользователю. Не принимать решение автоматически.

## Файлы

- **Читает:** `specs/**`, `git log` (для понимания темпа)
- **Пишет:** `specs/scenarios/<slug>.md`

## Правила

- **Три ветки — потолок.** Если больше — вопрос сформулирован слишком широко, разбейте.
- **Canary должен быть измерим.** «Если будет плохо» — не canary. «Если >3 scenarios требуют rewrite за 2 недели» — canary.
- **Status `superseded` — нормальное состояние.** Старые решения не удаляются, а отмечаются как перекрытые новыми.
- **Scenario-файл живёт один раз.** После `status: decided` — не редактируется; новое решение создаёт новый файл с `supersedes: <slug>`.
