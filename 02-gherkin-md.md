# Gherkin-MD как исполняемый L2

Gherkin-MD — это Markdown-документ с встроенными ```` ```gherkin ```` блоками. Человек читает как обычный Markdown; раннер (cucumber-js, behave, godog, SpecFlow, Reqnroll) извлекает fenced-блоки и исполняет их как BDD-тесты.

В этом справочнике Gherkin-MD — **рекомендованный диалект L2**. Он покрывает ключевое требование SDD: сценарии поведения должны быть одновременно (1) читаемы бизнесом, (2) исполняемы автоматически, (3) инвариантны к языку реализации.

---

## 1. Gherkin как оптимизированный промпт

Ключевая мысль SDD: **Gherkin-файл — это few-shot промпт в естественном языке, оптимизированный по конструкции**. Структура `Given / When / Then` не случайно совпадает с антологией промптов для LLM:

| Элемент Gherkin | Роль в промпте для AI | Что это даёт модели |
|---|---|---|
| `Feature` + описание | **System prompt** — задаёт домен и контекст | понимает область задачи |
| `Rule` | **Constraint** — бизнес-инвариант | не нарушает доменные правила |
| `Background` | **Context setup** — общие предусловия | не переизобретает состояние |
| `Given` | **Input state** — начальное состояние | знает, из чего исходить |
| `When` | **Action / trigger** — что реализовать | знает, *что* написать |
| `Then` | **Expected output** — критерий приёмки | знает, *как проверить* результат |
| `Scenario Outline` + `Examples` | **Few-shot examples** — пары вход/выход | получает обучающие примеры |
| Теги (`@unit`, `@integration`, `@billing`) | **Task metadata** — уровень, сервис, приоритет | выбирает стратегию реализации |

Следствия для SDD:

1. **Снятие двусмысленности до генерации кода.** Каждый `Then` — проверяемое утверждение. LLM работает с точностью контракта, а не с точностью угадывания.
2. **Авто-верификация сгенерированного кода.** `.feature`-файл *сам себя* проверяет через раннер. Цикл «написать код → прогнать сценарий → получить feedback» превращает vibe coding в управляемый цикл.
3. **Параллельная генерация.** Каждый `.feature` автономен. 10 файлов = 10 независимых задач для агентов, работающих одновременно без конфликтов.
4. **Инвариантность к языку.** Те же `.feature` работают с godog (Go), cucumber-js (Node), behave (Python), SpecFlow (.NET). При миграции стека меняются только step definitions.
5. **Контекст для code review.** Ревьюер (человек или модель) проверяет *соответствие кода сценарию*, а не угадывает намерение автора.
6. **Накопительная база знаний.** `specs/` со временем становится исполняемой документацией проекта: вечно актуальной (иначе CI падает) и переиспользуемой как библиотека промптов.

> «Когда AI генерирует код быстрее, чем человек печатает, узким местом становится качество спецификации.»

## 2. Когда выбирать Gherkin-MD

✅ **Выбирайте, если:**

- L2 должен быть исполняемым, а не просто документом
- Нужна читаемость для не-разработчиков (PM, аналитики, QA)
- В стеке есть Cucumber-совместимый раннер
- Хочется версионировать сценарии рядом с кодом, не в Jira
- Команда готова писать сценарии **до** кода

❌ **Не выбирайте, если:**

- Команда не готова поддерживать step definitions (glue-код)
- Сценарии покрывают низкоуровневые детали — используйте юнит-тесты напрямую
- Проект в фазе «рисую эскизы» — Markdown без fences временно достаточен

## 3. Структура файла

```markdown
---
version: 0.2.0
depends_on:
  constitution: 1.0.0
  domain: 0.3.0
status: active
feature_area: billing
---

# Feature: Выставление счёта

## FEA-001 — Выставление счёта

**Narrative:**
- **As a** оператор биллинга
- **I want** переводить подготовленный счёт в статус `issued`
- **So that** контрагент получает обязательство к оплате

### SCN-001 — Успешное выставление непустого счёта

- priority: MUST
- tags: [@smoke, @billing]
- references: { entities: [ENT-001, ENT-002], invariants: [INV-001], articles: [ART-003] }

​```gherkin
Feature: Issue invoice

  Rule: Empty invoices cannot be issued

    Scenario: Successful issuance of a non-empty invoice
      Given Invoice "INV-2026-000001" has at least one InvoiceLine
      When the operator issues Invoice "INV-2026-000001"
      Then Invoice "INV-2026-000001" status becomes "issued"
      And a domain event "InvoiceIssued" is emitted
      And no outbound network call is performed
​```
```

Три правила компоновки:

1. **Один fenced-блок — один Feature.** Раннер парсит содержимое блока как законченный `.feature`-файл.
2. **Метаданные сценария живут вне блока.** Priority, ссылки, теги — в Markdown-списке рядом. Раннер их не видит, человек и модель — видят.
3. **Rule опционален, но полезен** для группировки сценариев вокруг одного инварианта (см. §4).

## 4. Маппинг L2 → Gherkin

| Поле L2 | Конструкция Gherkin |
|---|---|
| `feature.narrative` | `Feature:` + `As a / I want / So that` |
| `feature.background` | `Background:` |
| Группа инвариантов (`INV-NNN`) | **`Rule:`** (см. ниже) |
| `scenario.given` | `Given` / `And` / `But` |
| `scenario.when` | `When` (**ровно один**) |
| `scenario.then` | `Then` / `And` / `But` |
| `scenario.examples` | `Scenario Outline:` + `Examples:` |
| `scenario.tags` | `@tag` над `Scenario:` |
| Структурные данные | `Data Tables` (см. §6) |
| Многострочный контент | `Doc Strings` (см. §6) |

### 4.1 Rule — группировка вокруг инварианта

`Rule:` появился в Gherkin v6. Позволяет сгруппировать сценарии, иллюстрирующие одно бизнес-правило — как раз то, что у нас в L1 называется `INV-NNN`. Использование:

```gherkin
Feature: Invoice issuance

  Rule: Empty invoice cannot be issued (INV-001)

    Scenario: Issue is rejected when zero lines
      Given Invoice "INV-2026-000001" has zero lines
      When the operator issues Invoice "INV-2026-000001"
      Then the action is rejected with reason "empty-invoice"

  Rule: Total must equal sum of lines (INV-002)

    Scenario: Total is recomputed when a line is added
      Given Invoice "INV-2026-000001" draft with total 0
      When an InvoiceLine with unitPrice 100 EUR and quantity 2 is added
      Then Invoice "INV-2026-000001" total becomes 200 EUR
```

**Правило:** каждый `Rule:` цитирует `INV-NNN` прямо в заголовке. Это делает связь L1↔L2 видимой раннеру (читает заголовок как текст) и человеку (видит в отчёте).

**`Rule` может содержать собственный `Background:`** — полезно, когда группа сценариев требует общих предусловий помимо Feature-wide.

### 4.2 Теги дублируются

Теги Markdown (в списке метаданных над блоком) — для человека и модели; `@tag` внутри Gherkin-блока — для раннера. Это намеренная избыточность: Markdown parser и Cucumber parser — разные инструменты, их нельзя унифицировать.

## 5. Русский Gherkin

Gherkin поддерживает 70+ языков. Для русскоязычных команд это снимает языковой барьер в командах с не-разработчиками (бизнес, аналитика, QA).

Активация — директивой в начале `.feature`-блока:

```gherkin
# language: ru

Функционал: Выставление счёта

  Правило: Пустой счёт нельзя выставить

    Сценарий: Отказ при нуле позиций
      Допустим счёт "INV-2026-000001" содержит 0 позиций
      Когда оператор выставляет счёт "INV-2026-000001"
      Тогда действие отклонено с причиной "empty-invoice"
```

### 5.1 Таблица ключевых слов

| English | Русский (канонический) | Альтернативы |
|---|---|---|
| `Feature` | `Функционал` | `Фича`, `Свойство`, `Функция` |
| `Rule` | `Правило` | — |
| `Background` | `Предыстория` | `Контекст` |
| `Scenario` | `Сценарий` | — |
| `Scenario Outline` | `Структура сценария` | — |
| `Examples` | `Примеры` | `Значения` |
| `Given` | `Допустим` | `Дано`, `Пусть`, `К тому же` |
| `When` | `Когда` | `Если` |
| `Then` | `Тогда` | `То` |
| `And` | `И` | `К тому же`, `Также` |
| `But` | `Но` | `А`, `Иначе` |

### 5.2 Когда русский, когда английский

| Критерий | Русский | Английский |
|---|---|---|
| Русскоязычная команда полностью | ✅ | ❌ |
| Нет-разработчики участвуют в написании | ✅ | ❌ |
| Международная команда | ❌ | ✅ |
| OSS-проект | ❌ | ✅ |
| Термины L1 только на английском | нейтрально | ✅ |
| Интеграция с English-only Cucumber fixtures | ❌ | ✅ |

**Рекомендация по умолчанию для этого справочника:** русский Gherkin для проектов с русскоязычной командой; английский — для международных и OSS.

**Никогда не смешивайте** русские и английские ключевые слова в одном файле. Cucumber-парсер читает `# language:` директиву и переключает таблицу ключевых слов — смесь приведёт к ошибкам парсинга.

## 6. Аргументы шагов

Gherkin поддерживает два механизма передачи структурированных данных в шаг: **Data Tables** (компактные табличные данные) и **Doc Strings** (многострочный текст).

### 6.1 Data Tables

Передаются в шаг как последний аргумент. Подходят для списков объектов, пар ключ-значение, матриц.

```gherkin
Scenario: Bulk invoice creation
  Given the following invoices exist:
    | number            | total  | currency | status |
    | INV-2026-000001   | 100.00 | EUR      | draft  |
    | INV-2026-000002   | 250.50 | EUR      | issued |
    | INV-2026-000003   |  75.00 | USD      | paid   |
  When the operator filters by currency "EUR"
  Then 2 invoices are returned
```

**Экранирование в ячейках:**

| Символ | Экранирование |
|---|---|
| перенос строки | `\n` |
| вертикальная черта | `\|` |
| обратный слэш | `\\` |

**Формы в step definitions:**

- `List<List<String>>` — список списков
- `List<Map<String, String>>` — список словарей (заголовки = ключи)
- `Map<String, String>` — пары ключ-значение (2 колонки)
- Кастомные трансформеры в доменные типы

### 6.2 Doc Strings

Многострочный текст, ограниченный тройными кавычками `"""` или тройными backticks. Подходят для JSON payloads, markdown, XML, SQL.

```gherkin
Scenario: Submit invoice webhook
  When the client posts to "/api/invoices" with body:
    """json
    {
      "number": "INV-2026-000001",
      "total": { "amount": 100.00, "currency": "EUR" },
      "lines": [
        { "description": "Consulting", "quantity": 1, "unitPrice": 100.00 }
      ]
    }
    """
  Then the server responds with status 201
```

Тип содержимого (`json`, `markdown`, `xml`) после кавычек — опциональная аннотация для подсветки и документации. Раннеру она безразлична, но делает `.feature` читаемее.

### 6.3 Когда что

| Данные | Механизм | Пример |
|---|---|---|
| Список объектов одного типа | Data Table | товары корзины, пользователи, счета |
| Пары ключ-значение | Data Table (2 колонки) | конфигурация, параметры |
| Параметры сценария × несколько прогонов | Scenario Outline + Examples | валидация возраста, ставки скидки |
| JSON payload | Doc String | тело HTTP-запроса, webhook |
| Markdown / длинный текст | Doc String | содержимое статьи, e-mail |
| Однострочное значение | аргумент шага (`"..."`) | имя, сумма, статус |

**Не путайте** `Scenario Outline + Examples` (параметризация — прогоняется N раз) и `Data Table` (одноразовый аргумент шага).

### 6.4 Звёздочка `*` как универсальный шаг

`*` — замена `Given`/`When`/`Then`/`And`/`But`, читаемая нейтрально. Полезна для перечислений:

```gherkin
Scenario: Default invoice bundle
  Given the system has a default invoice template with:
    * header section
    * line items block
    * totals footer
    * legal disclaimer
  When a new invoice is created from the template
  Then all four sections are present
```

## 7. Теги и фильтры

Теги начинаются с `@`. Ставятся перед `Feature`, `Rule`, `Scenario`, `Scenario Outline` или `Examples`. Наследуются от родителя к потомку.

### 7.1 Типичные категории

| Категория | Примеры | Назначение |
|---|---|---|
| Уровень тестирования | `@unit`, `@integration`, `@e2e` | фильтрация по уровню |
| Приоритет | `@smoke`, `@critical`, `@regression` | выбор набора для CI/CD |
| Сервис / домен | `@billing`, `@auth`, `@catalog` | группировка по bounded context |
| Состояние | `@wip`, `@skip`, `@manual` | управление выполнением |
| Среда | `@staging`, `@prod-safe` | контроль окружения |
| Трассировка | `@JIRA-1234`, `@sprint-42` | связь с задачами / эпиками |
| Атрибут поведения | `@negative`, `@happy-path`, `@edge-case` | характер сценария |

### 7.2 Выражения фильтрации

Cucumber/godog поддерживают логические операторы в `--tags`:

```bash
# Только smoke-тесты биллинга
godog --tags "@smoke and @billing"

# Всё кроме медленных
godog --tags "not @slow"

# Smoke ИЛИ critical
godog --tags "@smoke or @critical"

# Regression, но не wip и не skip
godog --tags "@regression and not @wip and not @skip"
```

### 7.3 Правила именования

- Нижний регистр, слова через дефис: `@order-management`
- Без пробелов (запрещены)
- Короткие и ёмкие: `@billing`, не `@billing-related-scenarios`
- Без дубликатов семантики: выберите `@smoke` ИЛИ `@critical`, не оба как синонимы

## 8. Лимиты и стиль

Жёсткие численные границы, которые делают сценарии атомарными и читаемыми:

| Лимит | Значение | Причина |
|---|---|---|
| Шагов на сценарий | **3–7** (максимум 10) | Длиннее = две поведения в одном сценарии |
| Сценариев на файл | **≤ 12** | Больше — файл становится непрочитываемым |
| Строк в Background | **≤ 4** | Больше — читатель не удержит контекст при чтении сценариев |
| Длина строки | ≤ 120 символов | Читаемость в diff-вьювере |
| Отступ | 2 пробела | Стандарт Gherkin |
| `When` на сценарий | **ровно 1** | Множественный `When` = множественное поведение |

**Декларативный стиль:** описывайте *что*, а не *как*.

```gherkin
# ❌ Императивно — хрупко и нечитаемо бизнесом
Scenario: Login
  Given user opens "/login"
  When user types "anna" in field "username"
  And user types "pass123" in field "password"
  And user clicks button "Sign in"
  Then user sees element ".dashboard-header"

# ✅ Декларативно — стабильно и ясно
Scenario: Successful login
  Given user "anna" is registered
  When user signs in with valid credentials
  Then the dashboard is displayed
```

**Говорящие имена** в Background и Given — «Anna Ivanova», «ACME Ltd», не «User A», «Customer X». Человеческий мозг лучше запоминает истории, чем обезличенные идентификаторы.

## 9. Определения шагов (step definitions)

Step definitions — «клей», связывающий фразы сценария с кодом. Правила:

1. **Одна фраза — одна реализация.** Двусмысленные шаги запрещены.
2. **Термины L1 дословно.** Если в L1 сущность — `Invoice`, в шаге — `Invoice`, не `Bill` и не `Счёт`.
3. **Step не знает про БД/HTTP.** Он вызывает доменный сервис или его порт. Ассерты — на наблюдаемое (возврат, событие, состояние агрегата), не на строки в таблицах.
4. **Файл step definitions цитирует `FEA-NNN` в заголовке** — для трассируемости.
5. **Организация — по домену, не по feature-файлам.** Иначе step definitions дублируются между файлами, ломая reuse.

Пример (JavaScript / cucumber-js):

```js
// features/steps/billing.steps.js
// Covers: FEA-001 (Invoice issuance)
import { Given, When, Then } from "@cucumber/cucumber";
import { Invoice } from "../../domain/ENT-001.Invoice"; // сущность из L1
import { issueInvoice } from "../../domain/invoice-service";

// INV-001, INV-002
Given("Invoice {string} has at least one InvoiceLine", function (n) {
  this.invoice = Invoice.parse({
    number: n,
    status: "draft",
    total: { amount: 100, currency: "EUR" },
    lines: [{ lineNumber: 1, description: "Consulting", quantity: 1, unitPrice: { amount: 100, currency: "EUR" } }],
    counterpartyId: "C-100",
  });
});

When("the operator issues Invoice {string}", function () {
  this.result = issueInvoice(this.invoice);
});

Then("Invoice {string} status becomes {string}", function (_, expected) {
  if (this.result.status !== expected) {
    throw new Error(`INV-001 violation: expected ${expected}, got ${this.result.status}`);
  }
});
```

**Сообщения ошибок цитируют `INV-NNN`** — падающий тест сразу указывает на нарушенный инвариант.

## 10. Антипаттерны

### 10.1 Процесс

- ❌ **Сценарии после кода.** Gherkin теряет основную цель — проверку общего понимания *до* разработки. Писать `.feature` всегда первым.
- ❌ **Единственный автор.** Сценарии, написанные в одиночку, не отражают общее понимание команды. Практика «Three Amigos» (бизнес + разработчик + тестировщик) — см. §12.
- ❌ **Feature-coupled step definitions.** Шаги, привязанные к конкретному `.feature`-файлу. Ведёт к дублированию. Организуйте по доменным понятиям.

### 10.2 Структура сценария

- ❌ **Несколько `When` в одном сценарии** (`When … And then I do …`). Два действия = два сценария.
- ❌ **Conjunction steps.** `Given I'm logged in and added item and went to cart` — один шаг делает слишком много. Разбить атомарно через `And`.
- ❌ **Зависимые сценарии.** Сценарий B использует состояние сценария A. Каждый должен создавать собственный контекст через `Given`.
- ❌ **Длинные сценарии** (>10 шагов). Обычно — тестирует несколько поведений.
- ❌ **Абстрактные двух-строчные сценарии** без конкретики. Нечего верифицировать.

### 10.3 Содержимое шагов

- ❌ **UI-детали** в сценариях: CSS-селекторы, XPath, имена кнопок. UI меняется чаще бизнес-логики, ломая тесты. Тестировать через API / бизнес-слой.
- ❌ **SQL и имена таблиц** в шагах. Это утечка реализации.
- ❌ **Глаголы взаимодействия в `Given`.** `Given` описывает факты, не действия. Для действий — `When`.
- ❌ **Размытые `Then`:** «всё работает», «ошибок нет». Укажите наблюдаемый признак успеха.
- ❌ **Императивные шаги** («клик по кнопке», «ввод в поле»). Декларативно: «оформляет заказ», «вводит корректные учётные данные».

### 10.4 Композиция файла

- ❌ **Один блок с десятком Feature.** Один fenced-блок — один Feature.
- ❌ **Step definitions, дёргающие БД/HTTP.** Тест превращается в интеграционный, теряется скорость и детерминизм.
- ❌ **Параметризация через Examples там, где нужны разные поведения.** Если у разных строк таблицы разный `Then` — это разные сценарии.
- ❌ **Regex-шаги, ловящие разные смыслы.** `Given (.*) exists` матчит всё подряд, step definition не знает, что делать.

## 11. Место Gherkin в пирамиде тестов

Пирамида тестирования: много юнит-тестов, умеренно интеграционных, мало сквозных. Gherkin наиболее эффективен на **интеграционном** и **приёмочном** уровнях; на юнит-уровне — в трёх конкретных случаях.

| Уровень | Скорость | Хрупкость | Ценность Gherkin |
|---|---|---|---|
| **Unit** | ~мс | низкая | средняя — только критичные бизнес-правила (см. ниже) |
| **Integration** | ~сек | средняя | **высокая — основная территория Gherkin** |
| **E2E / приёмочный** | ~мин | высокая | средняя — только ключевые smoke-сценарии |

### 11.1 Gherkin на уровне unit — когда оправдан

1. **Критичные бизнес-алгоритмы.** Расчёт скидок, налогов, комиссий — когда бизнес имеет мнение о результате и должен его подтверждать. Пример: шкала накопительной скидки в таблице `Examples` — бухгалтер читает и подтверждает, не заглядывая в код.
2. **Конечные автоматы** (state machines). `Invoice.status: draft → issued → paid` — каждый переход и запрещённый переход становится отдельным сценарием. Невидимая логика превращается в читаемый документ.
3. **Domain-level валидация** (инварианты `INV-NNN`). Нарушение инварианта — отдельный `Scenario`, явный `Then` с ожидаемым отказом.

### 11.2 Gherkin на интеграционном уровне — главная территория

Здесь Gherkin раскрывается максимально:

- **Общий язык для сложных взаимодействий.** API + БД + очереди + внешние сервисы — тест становится нечитаемым, если это код. Gherkin сохраняет понятность.
- **Раннее обнаружение интеграционных сюрпризов.** Изменения в формате ответа партнёрского сервиса ловятся до staging, с ясным сообщением *какое* поведение нарушено.
- **Переиспользование шагов между уровнями.** `Given product "SKU-100" exists in stock` в unit делает мок, в integration — вставку в тестовую БД, в e2e — вызов API. Сценарий не меняется, step definition меняется.
- **Документация межсервисных контрактов.** Новый разработчик читает `.feature` и понимает протокол взаимодействия без погружения в код двух сервисов.

### 11.3 Gherkin на E2E — только smoke

E2E-тесты медленные и хрупкие. Gherkin здесь полезен только для **критического пути** («пользователь регистрируется → оформляет заказ → оплачивает → получает подтверждение») — и одного-двух негативных smoke-сценариев.

### 11.4 Behaviour coverage ≠ code coverage

Традиционная метрика — `code coverage` (% строк кода). Gherkin даёт **`behaviour coverage`** — % бизнес-правил, имеющих покрывающий сценарий. Эта метрика значительно полезнее для бизнеса и менеджмента: видно, какие *правила* не тестируются, а не какие *строки*.

## 12. Инструменты и Three Amigos

### 12.1 Экосистема

| Инструмент | Язык / платформа | Статус |
|---|---|---|
| **Cucumber** (эталон) | Java, JS, Ruby | stable, широко распространён |
| **godog** | Go | официальный Cucumber for Go |
| **cucumber-js** | JavaScript / TypeScript | stable |
| **behave** | Python | lightweight |
| **pytest-bdd** | Python | как плагин к pytest |
| **SpecFlow / Reqnroll** | .NET / C# | SpecFlow deprecated, Reqnroll — преемник |
| **playwright-bdd** | JS/TS | BDD поверх Playwright для UI |
| **Karate** | Java | API-тестирование с Gherkin-синтаксисом |
| **gherkin-lint** | JS | линтер `.feature`-файлов |
| **Specmatic** | JVM | контрактное тестирование по OpenAPI/Gherkin |

### 12.2 Three Amigos

Практика совместного написания сценариев тремя ролями:

| Роль | Вклад |
|---|---|
| **Бизнес** (PO / аналитик) | определяет *что* и *зачем* |
| **Разработчик** | оценивает реализуемость, предлагает edge cases |
| **Тестировщик** | выявляет граничные условия и негативные сценарии |

Результат сессии Three Amigos — набор `.feature`-файлов, отражающих **общее понимание команды**, а не мнение одного автора. Это главный антидот антипаттерну «Single author».

Для SDD-команд на AI-ассистентах Three Amigos переформулируется как: **человек-бизнес + человек-разработчик + модель** (или **человек-продукт + модель в роли разработчика + модель в роли тестировщика**).

## 13. Связь с Dry Run

Перед генерацией кода модель обязана проверить:

- [ ] Все имена сущностей в шагах совпадают с L1 дословно
- [ ] Каждый сценарий ссылается хотя бы на один `ENT-NNN` или `INV-NNN`
- [ ] В каждом сценарии **ровно один** `When`
- [ ] Количество шагов на сценарий в диапазоне 3–10
- [ ] Background не длиннее 4 строк
- [ ] Нет UI-деталей, SQL, имён таблиц
- [ ] Нет шагов, требующих поведения, отсутствующего в L2
- [ ] Версии `depends_on` совместимы с текущими L0 и L1
- [ ] Если используется `Rule:` — он цитирует `INV-NNN` в заголовке

Если хоть один пункт не выполнен — остановка и эскалация через `/sdd:dry-run` Verdict: `BLOCK`.

Канонический протокол Dry Run — [`04-golden-rules.md` → Rule 3](04-golden-rules.md#rule-3--dry-run--сверка-перед-кодингом).
