# Глоссарий SDD-справочника

Термины, которые использует этот справочник. Определения каноничны: при расхождении в других документах побеждает определение отсюда.

---

### Aggregate / Агрегат

Кластер связанных сущностей с единой границей консистентности и корневой сущностью (Aggregate Root). Внешний код обращается к агрегату только через корень; внутренние сущности агрегата изменяются атомарно. Термин из Domain-Driven Design.

### Aggregate Root / Корень агрегата

Сущность, через которую внешний мир взаимодействует с агрегатом. Все инварианты агрегата проверяются корнем. Только корень имеет внешний идентификатор, по которому на агрегат можно сослаться.

### Bounded Context / Ограниченный контекст

Часть системы с собственной доменной моделью и единым языком. Внутри bounded context один и тот же термин имеет одно и то же значение. Между контекстами возможны синонимы (напр., `Customer` в billing и в CRM — разные сущности).

### BDD (Behavior-Driven Development)

Подход, описывающий поведение системы в формате Given/When/Then. В SDD эти сценарии живут на уровне L2.

### Dialect / Диалект

Синтаксис, в котором записана спецификация (Markdown, YAML, Gherkin-MD, JSON Schema, Pkl, CUE, …). Выбор диалекта определяет степень исполнимости спеки.

### Domain Event / Доменное событие

Факт, который произошёл в домене и имеет бизнес-значение (`OrderPlaced`, `PaymentReceived`). Именуется в прошедшем времени. Живёт на уровне L1.

### Context rot / Деградация контекста

Снижение качества вывода LLM по мере заполнения контекстного окна reasoning trail'ом. Симптомы: модель «забывает» правила из L0, путает версии сущностей, теряет приоритеты pinned-блоков. Ловится по простому маркеру: контекст > 60% бюджета окна → пора рестартовать. См. [`03-context-injection.md` § 9](03-context-injection.md#9-context-rot-и-phase-restart). Заимствовано из [GSD context engineering](https://github.com/gsd-build/get-shit-done).

### Dry Run

Обязательная сверка задачи со спецификациями до генерации кода. Модель перечисляет затронутые `ART`/`ENT`/`INV`/`SCN`, проверяет конфликты с L0, сообщает о недостающих спеках. См. [`04-golden-rules.md` → Rule 3](04-golden-rules.md#rule-3--dry-run--сверка-перед-кодингом).

### Entity / Сущность

Объект с идентичностью, существующий во времени. Два разных счёта с одинаковыми полями, но разными номерами — разные сущности. Противоположность — Value Object.

### Feature (L2)

Группа сценариев, описывающих одну capability. Имеет `FEA-NNN`, narrative в формате As/I want/So that и от одного до многих сценариев.

### Golden Rules / Золотые правила

Четыре нелокальных правила, нарушение которых ломает всю систему SDD: Atomic, SSOT, Dry Run, Stable IDs. См. [`04-golden-rules.md`](04-golden-rules.md).

### Invariant / Инвариант

Утверждение, которое всегда истинно для валидного состояния сущности. Выражается с RFC 2119 (`MUST`/`MUST NOT`). Нарушение = битое состояние = ошибка в коде.

### L0 Constitution / Конституция

Высший уровень спецификаций. Содержит неизменные законы проекта: tech-stack, архитектурные границы, требования безопасности, бюджеты производительности. Побеждает в любом конфликте с L1/L2.

### L1 Domain Model / Доменная модель

Средний уровень. Существительные системы: сущности, значения, агрегаты, инварианты, отношения, доменные события. Ссылается вверх на L0 через `constraints_from_L0`.

### L2 BDD Scenarios / Сценарии

Нижний спецификационный уровень. Глаголы системы: Given/When/Then-сценарии наблюдаемого поведения. Ссылается вверх на L1 (сущности/инварианты) и L0 (статьи).

### Maturity Ladder / Зрелостная шкала

Три уровня зрелости спек: spec-first (выкидывается), spec-anchored (живёт в репо), spec-as-source (код генерируется из спек). SDD-справочник описывает spec-anchored. См. [`04-golden-rules.md` → Maturity Ladder](04-golden-rules.md#бонус-maturity-ladder--зрелостная-шкала).

### RFC 2119

IETF-стандарт ключевых слов для нормативных документов: `MUST`, `MUST NOT`, `SHOULD`, `SHOULD NOT`, `MAY`. SDD требует, чтобы каждая нормативная формулировка (статьи L0, инварианты L1, priority сценариев L2) содержала одно из этих ключевых слов.

### SSOT (Single Source of Truth) / Единственный источник правды

У каждого факта ровно одно место определения; остальные — ссылки. См. [`04-golden-rules.md` → Rule 2](04-golden-rules.md#rule-2--single-source-of-truth--единственный-источник-правды).

### Pattern (core / satellite)

Операционное правило работы Claude со спеками. В отличие от содержимого спек, pattern описывает *процесс*, а не факты домена: «Dry Run перед кодом», «один `when` на сценарий». **Core patterns** (`memory/patterns.md`, hard cap 70 строк / 5.5 KB) загружаются каждой командой; **satellite patterns** (`memory/patterns/<context>.md`, soft cap 30 строк) — только когда задача касается соответствующего bounded context. См. [`06-patterns.md`](06-patterns.md).

### Pattern Routing

Принцип разделения operational rules: универсальные правила идут в core, доменно-специфичные — в satellite. Механика заимствована у [cog](https://github.com/marciopuga/cog) и адаптирована под SDD. Проверяется через `bin/sdd-validate.sh` (caps) и `bin/sdd-scorecard.sh` (метрики использования).

### Gate / Гейт

Точка проверки в процессе SDD, которая может остановить или замедлить движение задачи. Имеет четыре канонических типа: **pre-flight** (блокирует старт фазы), **revision** (просит доработать в процессе), **escalation** (передаёт неоднозначность человеку), **abort** (останавливает критически). См. [`04-golden-rules.md` → Гейты](04-golden-rules.md#гейты--таксономия-проверок). Заимствовано у [GSD](https://github.com/gsd-build/get-shit-done).

### Steering document / Документ направления

Persistent project context — в отличие от L0/L1/L2 (нормативные спеки), описывает «где мы есть»: vision продукта, выбранный стек, физическая раскладка кода. Загружается один раз при онбординге и цитируется (не дублируется) специками. Три канонических документа: `steering/product.md`, `steering/tech.md`, `steering/structure.md`. Шаблоны — в `templates/steering-*.tpl.md`. Заимствовано у [Pimzino spec-workflow](https://github.com/Pimzino/claude-code-spec-workflow).

### Lifecycle / Жизненный цикл спецификации

Каждый файл L0/L1/L2 имеет в frontmatter поле `status:` с одним из трёх значений:

- **`draft`** — спецификация в работе. Может содержать черновые формулировки, незаконченные секции, открытые вопросы. **Не обязательна** к соблюдению. Запрещается ссылаться на ID из draft-файла из active-файла. Pre-commit hook может предупреждать о коммите draft в main-ветку.
- **`active`** — спецификация введена в строй. Все её формулировки **обязательны** к соблюдению. Изменения требуют bump версии: patch для clarifications, minor для добавлений, major для изменений семантики (с ре-ревью всех L1/L2, см. процедуру в L0 §3.6).
- **`deprecated`** — спецификация заменена другой или признана устаревшей. **ID не переиспользуются** (Stable IDs, Rule 4). Файл остаётся в репо для исторической трассируемости. В заголовке должна быть ссылка на заменяющую спецификацию (`supersedes:` field или явный комментарий).

**Переходы:** `draft → active` через ревью; `active → deprecated` при замене или удалении; обратные переходы запрещены (восстановление через новую версию).

### Phase restart / Рестарт фазы

Переход к следующей фазе SDD-процесса (Dry Run → Plan → Code → Verify) **в свежем контекстном окне**, передавая на вход только финальные артефакты предыдущей фазы, а не reasoning trail. Главное противоядие [context rot](#context-rot--деградация-контекста). См. [`03-context-injection.md` § 9.2](03-context-injection.md#92-стратегия-phase-restart-с-артефактами).

### Second opinion / Второе мнение

Независимая верификация результата `/sdd:dry-run` через **sub-agent с чистым контекстом**. Защита от self-confirmation bias: одна и та же модель в одной и той же сессии может незаметно убедить себя, что её Dry Run верен. Реализуется через Task tool Claude Code. См. [`/sdd:second-opinion`](commands/second-opinion.md). Заимствовано из [`tylerburleigh/claude-sdd-toolkit`](https://github.com/tylerburleigh/claude-sdd-toolkit) (multi-model consultation), адаптировано под one-model setup через context isolation.

### Slice / Срез

Подмножество L1 или L2, подаваемое модели в контекст. Собирается из задачи через seed + транзитивное замыкание. См. [`03-context-injection.md` → §4](03-context-injection.md#4-стратегии-срезов).

### Stable Identifier / Стабильный идентификатор

ID (`ART-NNN`, `ENT-NNN`, `INV-NNN`, `EVT-NNN`, `FEA-NNN`, `SCN-NNN`), который никогда не переиспользуется, даже после удаления соответствующего элемента. См. [`04-golden-rules.md` → Rule 4](04-golden-rules.md#rule-4--stable-identifiers--стабильные-идентификаторы).

### State Machine / Машина состояний

Опциональная часть описания сущности L1: набор состояний, начальное состояние, терминальные состояния и разрешённые переходы.

### Value Object / Значение

Объект без идентичности, определяемый только значениями атрибутов. Два `Money(100, EUR)` — это одно и то же значение. Value Objects неизменяемы (immutable) по определению.

### Version Handshake / Версионная сверка

Проверка совместимости версий уровней: `depends_on.constitution` в L1/L2 ДОЛЖЕН совпадать с текущей версией L0. Входит в Dry Run. См. [`03-context-injection.md` → §7](03-context-injection.md#7-версионный-handshake).
