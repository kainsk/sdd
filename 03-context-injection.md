# Context Injection — как скармливать спецификации модели

Спецификация, которую модель не видит в момент генерации, не существует. Техника Context Injection отвечает на вопросы: *что* подавать, *в каком порядке*, *сколько* и *как гарантировать*, что модель не проигнорирует.

---

## 1. Принципы

1. **L0 всегда целиком, дословно.** Конституция не сокращается никогда. Её объём — верхний предел, и он ограничивает сам размер Конституции: если L0 не помещается в стартовый бюджет — Конституция слишком большая.
2. **L1 срезом по релевантности.** Полная доменная модель обычно избыточна. В контекст попадают только сущности, упомянутые в задаче, плюс их прямые зависимости (`relations.target`, типы полей).
3. **L2 срезом по тегам/feature_area.** Сценарии выбираются по совпадению с задачей.
4. **Ничего не summarize.** Сжатое в прозу L0 — это другое L0. Сокращения делаются удалением несвязанных частей, а не перефразированием.
5. **Порядок фиксирован.** Модель читает сверху вниз, и первые токены весят больше. L0 идёт первым всегда.
6. **Dry Run обязателен.** Перед любым кодингом модель обязана сверить задачу со спецификациями и отчитаться о противоречиях. Канонический протокол — [`04-golden-rules.md` → Rule 3](04-golden-rules.md#rule-3--dry-run--сверка-перед-кодингом). Ниже — только интеграция Dry Run в инъекцию.

## 2. Порядок инъекции

```
┌─────────────────────────────────────────────┐
│  SYSTEM PROMPT                              │
│  ┌───────────────────────────────────────┐  │
│  │ 1. Role & instructions                │  │
│  │ 2. Steering: product / tech / structure│ │  ← pinned (если есть)
│  │ 3. L0 Constitution (verbatim, full)   │  │  ← pinned
│  │ 4. Glossary                           │  │  ← pinned
│  │ 5. Patterns (core + relevant satellites)│  ← pinned
│  │ 6. L1 slice (relevant entities)       │  │
│  │ 7. L2 slice (relevant scenarios)      │  │
│  │ 8. Dry Run protocol                   │  │  ← pinned
│  └───────────────────────────────────────┘  │
├─────────────────────────────────────────────┤
│  USER PROMPT                                │
│  ┌───────────────────────────────────────┐  │
│  │ 9. Task description                   │  │
│  │ 10. (optional) code context snippets  │  │
│  └───────────────────────────────────────┘  │
└─────────────────────────────────────────────┘
```

**Pinned** = не урезается ни при каких условиях. Если бюджета не хватает — урезаются L1/L2 срезы или code snippets, но не pinned-блоки.

**Steering documents (опционально, но желательно):** загружаются вторым блоком после role. Это `steering/product.md` (vision/users), `steering/tech.md` (стек), `steering/structure.md` (раскладка). В отличие от L0 — не нормативные, описывают «где мы есть». Если steering-файлы отсутствуют (early-stage project) — блок пропускается. Если присутствуют — загружаются **целиком, всегда**, не урезаются: они короткие (≤ 2KB каждый) и задают абсолютный контекст.

## 3. Бюджет контекста

Ориентировочное распределение для задачи среднего размера (одна capability, 3–10 сценариев):

| Блок | Доля бюджета | Примечание |
|---|---|---|
| Role & instructions | 2–5% | Короткая, реюзабельная |
| Steering (product+tech+structure) | 2–5% | Опционально; полностью или вовсе нет |
| L0 Constitution | 5–15% | Если > 15% — Конституция раздута |
| Glossary | 1–3% | Только термины, на которые ссылаются инварианты |
| Patterns (core + satellites) | 1–3% | Core hard cap 5.5KB → ~2% |
| L1 slice | 15–30% | Срез по релевантности |
| L2 slice | 10–25% | Срез по тегам |
| Dry Run protocol | 1–2% | Короткий, фиксированный |
| Task + code context | 20–35% | Остаток |

Если суммарно L0+L1+L2 перевалили за 60% — значит задача слишком крупная для одного промпта. Разделите.

## 4. Стратегии срезов

### 4.1 L1 слайсинг

Срез L1 строится за три шага:

1. **Seed** — сущности, явно упомянутые в задаче по имени.
2. **Transitive closure по relations** — добавить всё, на что seed ссылается через `relations.target` (глубина 1 обычно достаточна; глубина 2 только если инварианты пересекают эту границу).
3. **Invariants transitive closure** — добавить все инварианты, у которых `entity` попал в срез, плюс инварианты, цитируемые в `cross_refs` из выбранных статей L0.

Результат — минимальный L1, в котором ни одна формулировка не ссылается на сущность/инвариант вне среза.

### 4.2 L2 слайсинг

1. **Seed** — сценарии, у которых `feature_area` или `tags` совпадают с задачей.
2. **Priority filter** — при нехватке бюджета сначала отбрасываются `MAY`, затем `SHOULD`. `MUST`-сценарии никогда не отбрасываются.
3. **Reference closure** — проверить, что все `references.entities` из оставшихся сценариев попали в срез L1. Если нет — добавить.

### 4.3 Признак плохого среза

- Срез L1 содержит сущность, упомянутую в `then` сценария, но отсутствующую в `references`
- В срезе L2 есть `SCN-NNN`, чей `priority: MAY` вытеснил `SCN-NNN` с `priority: MUST` — это ошибка фильтрации
- Срез ссылается на `ART-NNN`, которой нет в текущей версии L0 (версии разошлись)

При любом признаке — стоп, пересобрать срез.

## 5. Скелет system prompt

```text
# Роль
Ты генерируешь код для <проект>. Спецификации — твой единственный источник истины.

# Иерархия авторитета
L0 Конституция > L1 Доменная модель > L2 BDD-сценарии > твоё собственное суждение.
При любом конфликте двух источников — ОСТАНОВИСЬ и сообщи. Не пытайся удовлетворить оба.

# L0 — Конституция (дословно, авторитетно)
<...полное содержимое L0...>

# Глоссарий (авторитетно)
<...термины...>

# L1 — Срез доменной модели
Версии: constitution=<X>, domain=<Y>
<...релевантные сущности, value objects, инварианты, события...>

# L2 — Срез BDD-сценариев
Версии: constitution=<X>, domain=<Y>, scenarios=<Z>
<...релевантные Feature(s) и Scenario(s)...>

# Протокол Dry Run (обязателен перед любым кодом)
Перед генерацией реализации:
1. Перечисли каждый ART-NNN, ENT-NNN, INV-NNN, SCN-NNN, которого касается задача.
2. Проверь каждый на противоречие задаче или неоднозначность.
3. Если есть конфликт с L0 — СТОП. Процитируй конфликтующее правило дословно.
4. Если требуется сущность/сценарий, отсутствующий в срезе — СТОП. Запроси спеку.
5. Код допустим только после чистого Dry Run.

# Формат вывода
- Начни с секции "## Dry Run" с перечнем находок.
- Затем "## Plan" — короткий список шагов.
- Затем "## Code" — реализация.
- Никогда не пропускай секцию Dry Run.
```

Если рабочая модель стабильнее реагирует на англоязычные системные промпты — допустимо дословно перевести этот скелет на английский. Но тогда **все** пункты переводятся синхронно; смешанный промпт (половина EN, половина RU) запрещён — он снижает строгость следования правилам.

## 6. Dry Run: шаблон для пользователя

Когда пользователь хочет перепроверить уже выданный срез или существующий код, он вставляет этот блок в начало сообщения:

```text
Dry Run request.

Given the attached L0/L1/L2, analyze the following task/code and report:
1. Every spec reference (ART/ENT/INV/SCN) the task touches.
2. Any contradiction with L0 — verbatim quote of the conflicting rule.
3. Any required entity/scenario missing from L1/L2.
4. Any term used ambiguously or not defined in the glossary.

Do not produce code in this response. Report findings only.
```

Результат Dry Run — это не код, а список находок. Кодинг разрешён только после чистого прогона.

## 7. Версионный handshake

Каждый срез, попадающий в контекст, сопровождается версионной шапкой:

```yaml
spec_bundle:
  constitution: 1.0.0
  domain: 0.3.0
  scenarios: 0.2.0
  sliced_at: 2026-04-10
```

Модель инструктируется:

> «Перед Dry Run сверь, что `depends_on` внутри L1 и L2 совместим с версией L0 в этом бандле. Если версии расходятся — сообщи и не продолжай.»

Это ловит ситуации, когда L0 изменилась, а L1/L2 не ре-ревьюировались.

## 8. Параллельная генерация по независимым L2-файлам

Если L2 оформлен в Gherkin-MD (см. [`02-gherkin-md.md`](02-gherkin-md.md)), каждый `.feature`-файл — автономная единица работы. Это открывает **параллельную генерацию**:

```
specs/scenarios/
├── billing.md       ──→ агент #1 ──→ код billing       ✓ 5/5 сценариев
├── user-access.md   ──→ агент #2 ──→ код user-access   ✓ 3/3 сценария
├── catalog.md       ──→ агент #3 ──→ код catalog       ✓ 4/4 сценария
└── payment.md       ──→ агент #4 ──→ код payment       ✓ 2/2 сценария
```

Каждый агент:

1. Получает **свой** срез бандла: L0 (общий) + L1 (общий или слайс bounded context) + **один** L2-файл + задачу
2. Выполняет Dry Run на *своём* срезе
3. Генерирует код только для *своего* `.feature`
4. Результат верифицируется *своими* сценариями

**Условия безопасной параллелизации:**

- L0 действительно pinned (все агенты читают **одну и ту же** версию)
- Bounded contexts изолированы на уровне L1 — агенты не редактируют общие сущности
- Отсутствуют хот-инварианты, на которые ссылаются все feature (видно через `/sdd:foresight` → hot invariants)

**Когда не параллелить:**

- Один инвариант цитируется из 3+ feature-файлов → изменение кода под один сценарий может сломать другой
- Задача затрагивает несколько bounded contexts одновременно → сначала сделайте её последовательно, потом декомпозируйте

Gherkin как спецификация и как оптимизированный промпт — [`02-gherkin-md.md` § 1](02-gherkin-md.md#1-gherkin-как-оптимизированный-промпт).

## 9. Context rot и phase restart

**Context rot** — деградация качества вывода LLM по мере заполнения контекстного окна reasoning trail'ом. Феномен наблюдается во всех современных моделях с большим контекстом (Claude, GPT, Gemini): первые 50–80 тысяч токенов модель использует с высокой точностью, после ~100 тысяч начинается «прогрессивный туман» — модель помнит факты, но всё хуже соблюдает ограничения, путается в перекрёстных ссылках, теряет приоритеты pinned-блоков.

Заимствовано из [GSD context engineering](https://github.com/gsd-build/get-shit-done): «atomic context windows per task — fresh 200k-token context per phase to prevent context rot».

### 9.1 Симптомы context rot

- Модель начинает «забывать» правила из L0, хотя они всё ещё в контексте дословно
- Dry Run перестаёт ловить очевидные противоречия с инвариантами L1
- В одном ответе модель цитирует разные версии одной и той же сущности (как будто прочитала L1 дважды и получила два варианта)
- Растёт latency, снижается уверенность в формулировках
- Модель начинает «галлюцинировать» ссылки на ID, которых нет в спеках

### 9.2 Стратегия: phase restart с артефактами

**Принцип:** не давайте контексту копиться. Каждая логическая фаза работы (Dry Run → Plan → Code → Verify) должна стартовать в **свежем контекстном окне**, передавая на вход только **готовые артефакты предыдущей фазы**, а не полный reasoning trail.

```
Фаза 1 — Dry Run         Фаза 2 — Plan          Фаза 3 — Code
┌──────────────┐         ┌──────────────┐       ┌──────────────┐
│ L0 + L1+L2   │         │ L0 + L1+L2   │       │ L0 + L1+L2   │
│ patterns     │         │ patterns     │       │ patterns     │
│ task         │         │ Dry Run      │       │ Plan         │
│              │ ──art→  │ artifact     │ ──art→│ artifact     │
│ → ## Dry Run │         │ → ## Plan    │       │ → ## Code    │
└──────────────┘         └──────────────┘       └──────────────┘
   100k токенов            120k токенов           150k токенов
   но БЕЗ trail предыдущих фаз — только их финальные блоки
```

**Что является «артефактом»:**

- Dry Run → секция `## Dry Run` целиком (Touched / Contradictions / Missing / Verdict)
- Plan → секция `## Plan` (короткий список шагов)
- Code → diff или новые файлы

**Что НЕ передаётся между фазами:**

- Внутренний reasoning модели («я подумал… потом передумал…»)
- Промежуточные draft'ы кода
- Отброшенные варианты архитектуры
- Объяснения «почему я выбрал X вместо Y» — если важно, выносится в `decisions.md`

### 9.3 Research isolation

GSD выделяет отдельный паттерн — **research isolation**: когда задача требует исследования (грепнуть кодбазу, прочитать чужой код, изучить документацию), это **не должно засорять контекст исполнения**.

Структурный приём: исследование живёт в **отдельной фазе с отдельным контекстом**.

```
Research phase                   Execution phase
┌──────────────────┐             ┌──────────────────┐
│ L0+L1+L2         │             │ L0+L1+L2         │
│ task: "найти X,  │             │ patterns         │
│  понять Y"       │             │ task: "сделать Z"│
│                  │             │                  │
│ → research notes │ ──notes──→  │ + research notes │
│                  │             │                  │
└──────────────────┘             └──────────────────┘
   читает много,                    пишет код,
   пишет конспект                   не читает кодбазу заново
```

В Claude Code это легко реализуется через **sub-agents** (Task tool): research-агент работает в своём контексте, возвращает summary, основной агент использует summary без полного trail.

### 9.4 Когда применять phase restart

| Размер задачи | Стратегия |
|---|---|
| Тривиальная (5–10 минут модели) | Один проход, без рестарта |
| Средняя (1 чёткий фит → код → тест) | Рестарт между Dry Run и Code |
| Крупная (research + design + code) | Рестарт после каждой фазы (research → plan → code → verify) |
| Многодневная (фича из 10+ файлов) | Рестарт после каждого `.feature` сценария + параллельная генерация (см. §8) |

**Маркер для решения:** если контекст в текущей сессии перевалил за **60% бюджета окна** (например, ~120k из 200k), и впереди ещё работа — **немедленный рестарт** с компактификацией. Не «дотянем как-нибудь».

### 9.5 Связь со слайсингом

Phase restart **усиливает** слайсинг. Слайсинг урезает то, что подаётся ВНУТРИ одной фазы; phase restart урезает то, что **переходит между фазами**. Оба механизма работают вместе: каждая фаза получает свой минимально необходимый L1/L2 срез + только финальные артефакты предыдущей фазы.

Без phase restart слайсинг частично теряет смысл — модель «вспомнит» неурезанный контекст из reasoning trail прошлой фазы, и L1/L2 срезы окажутся бесполезными.

### 9.6 Антипаттерны

- ❌ **«Одна большая задача за один проход»** на 50+ шагов. Контекст забивается до того, как модель закончит.
- ❌ **Передача reasoning trail между фазами.** «Помни, я раньше думал…» — модель будет смешивать прошлые и текущие выводы.
- ❌ **Research внутри execution.** Греп и чтение кодбазы делается до фазы кода, не во время.
- ❌ **Игнорирование 60% маркера.** «Ещё одна-две задачи и закончим» — обычно заканчивается падением качества.

## 10. Когда SDD **не** применять

Не всякая задача требует полного каскада. Честный перечень ситуаций, когда SDD избыточен:

- **Мелкий баг**, затрагивающий уже существующий `SCN-NNN`/`INV-NNN` → используйте структурированный fast track [`/sdd:bug`](commands/bug.md) (Report → Analyze → Fix → Verify)
- **Правка текста / i18n** → код напрямую
- **Переименование переменной** без семантических изменений → рефакторинг
- **Чистый UI-тюнинг** (отступы, цвета) → код
- **Experiment / spike**, результат которого будет удалён → можно опустить L2

Граница простая: если изменение **может породить новый инвариант или сценарий** — нужен полный `/sdd:dry-run`. Если только восстанавливает соответствие коду уже существующих спек — `/sdd:bug` достаточно. Иначе — обычное редактирование.

Это прямое противоядие pitfall'у *workflow mismatch*, описанному в критике SDD: обязательный тяжёлый пайплайн для тривиальных задач убивает продуктивность быстрее, чем отсутствие спек. Bug fast track даёт **структурированный, но облегчённый** путь: есть отчёт, есть тест, есть верификация — но без генерации новых ID.

## 11. Антипаттерны инъекции

- ❌ Summarize L0 «чтобы помещалось»
- ❌ Подавать L2 без L1
- ❌ Подавать L1-срез, где сущность ссылается на отсутствующую в срезе
- ❌ Пропускать Dry Run «ради скорости»
- ❌ Класть L0 не в начало
- ❌ Забывать про версионный handshake
- ❌ Использовать один раздутый system prompt на все задачи — переиспользуется role, переинжектируются срезы
- ❌ Давать модели доступ к «полному репо спек» без слайсинга — это аналог `z.passthrough()`
